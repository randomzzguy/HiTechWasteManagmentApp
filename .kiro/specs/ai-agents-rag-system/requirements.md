# Requirements Document

## Introduction

The **AI Agents & RAG System** feature formalises and completes the intelligent automation layer of the Hi-Tech Waste Management platform. Five specialised AI agents — Compliance, ESG & Carbon, Operations & Scheduling, Fleet & Maintenance, and Client Intelligence — are already partially implemented as Celery tasks and FastAPI endpoints. This feature defines the complete requirements for all five agents, the RAG (Retrieval-Augmented Generation) pipeline that powers them, the agent event persistence and notification system, and the frontend components that surface agent outputs to platform users.

The system uses Ollama (llama3 for reasoning, nomic-embed-text for embeddings), Milvus v2.3 as the vector store, Celery + Redis for scheduled and event-driven task execution, and Server-Sent Events (SSE) for real-time streaming of AI responses to the frontend.

---

## Glossary

- **Agent**: A specialised AI module that monitors a domain of platform data, generates alerts and recommendations, and answers natural language queries within its domain.
- **RAG**: Retrieval-Augmented Generation — the technique of retrieving relevant document chunks from a vector store and injecting them into the LLM prompt to ground responses in factual platform data.
- **Agent_Event**: A persisted record of an alert, recommendation, report, or action generated by an agent, stored in the `agent_events` table.
- **Milvus_Collection**: A named vector store partition in Milvus that holds embedded document chunks for a specific scope (platform-wide or client-specific).
- **Chunk**: A segment of text extracted from a document, typically 800 words with 100-word overlap, that is embedded and stored in Milvus.
- **Embedding**: A 768-dimensional float vector produced by the `nomic-embed-text` model that represents the semantic meaning of a text chunk.
- **SSE**: Server-Sent Events — a unidirectional HTTP streaming protocol used to deliver LLM token output to the frontend in real time.
- **Intent_Detection**: The process of analysing a user's chat message to determine which agent should handle it, based on keyword scoring.
- **System_Prompt**: The instruction text prepended to every LLM conversation that defines the agent's persona, domain expertise, and response style.
- **Celery_Beat**: The Celery periodic task scheduler that triggers agent runs on a fixed schedule.
- **AgentAlertFeed**: The frontend component that displays a live feed of agent events, updated via WebSocket.
- **AgentStatusPanel**: The frontend component that shows the current run status and last-run time of each agent.

### Requirement 1: Compliance Agent — Scheduled Deadline Monitoring

**User Story:** As a Compliance_Officer or Operations_Manager, I want the Compliance Agent to automatically monitor scheduled waste storage deadlines and alert me before batches exceed the 90-day limit, so that I can arrange disposal in time and avoid regulatory penalties.

#### Acceptance Criteria

1. THE Compliance Agent SHALL run automatically every 6 hours via Celery Beat.
2. WHEN the Compliance Agent runs, THE System SHALL query all `scheduled_waste_batches` with `status = 'in_storage'` and `storage_deadline <= CURRENT_DATE + 14 days`.
3. FOR each batch within 14 days of its deadline, THE System SHALL create an `Agent_Event` with `event_type = 'alert'`, `severity = 'warning'`, and a body describing the SW code, client name, quantity, and days remaining.
4. FOR each batch that has exceeded its 90-day deadline (`days_remaining < 0`), THE System SHALL create an `Agent_Event` with `severity = 'critical'` and broadcast it to the WebSocket `agent-alerts` room.
5. WHEN a batch is overdue and the client has a registered PIC email, THE System SHALL send an email notification to the client's PIC email address.
6. THE Compliance Agent SHALL also be triggered event-driven when a new `scheduled_waste_batch` is created, in addition to its scheduled run.
7. THE Compliance Agent SHALL use the RAG pipeline to retrieve relevant regulatory context (SW codes, EQA Act 127, e-SWIS guide) when answering on-demand queries via the AI chat endpoint.

### Requirement 2: Compliance Agent — On-Demand Query Handling

**User Story:** As a Compliance_Officer, I want to ask the AI assistant questions about SW codes, packaging requirements, and consignment note procedures, so that I can get accurate regulatory guidance without searching through documents manually.

#### Acceptance Criteria

1. WHEN a user submits a chat message containing compliance-related keywords (e.g. "SW code", "consignment note", "e-SWIS", "90-day", "EQA"), THE System SHALL route the query to the Compliance Agent.
2. THE Compliance Agent SHALL retrieve up to 8 relevant chunks from the Milvus `hitech_rag` collection using the query embedding before generating a response.
3. THE Compliance Agent's system prompt SHALL identify it as a DOE scheduled waste compliance expert with knowledge of EQA 1974, Scheduled Wastes Regulations 2005, SW codes, and Cenviro procedures.
4. THE Compliance Agent SHALL cite specific regulation clauses or SW codes in its responses when available in the retrieved context.

### Requirement 3: ESG & Carbon Agent — Weekly Performance Analysis

**User Story:** As a Management user, I want the ESG Agent to automatically calculate weekly diversion rates and carbon impact per client and alert me when clients fall below their SLA targets, so that I can proactively address underperformance.

#### Acceptance Criteria

1. THE ESG Agent SHALL run automatically every Monday at 08:00 MST (00:00 UTC) via Celery Beat.
2. WHEN the ESG Agent runs, THE System SHALL aggregate the previous 7 days of `weighbridge_records`, `recyclable_records`, and `carbon_records` per active client.
3. FOR each client, THE System SHALL calculate the diversion rate as `(total_recyclable_kg / total_waste_kg) × 100`, rounded to 2 decimal places.
4. FOR each client whose diversion rate is below their `sla_diversion_target`, THE System SHALL create an `Agent_Event` with `event_type = 'alert'`, `severity = 'warning'`, and a body stating the actual rate, target rate, and gap.
5. THE ESG Agent SHALL generate a weekly summary narrative using the LLM and persist it as an `Agent_Event` with `event_type = 'report'` and `severity = 'info'`.
6. THE ESG Agent SHALL also be triggerable on-demand via the AI chat endpoint for specific client ESG queries.

### Requirement 5: Operations & Scheduling Agent — Daily Morning Briefing

**User Story:** As an Operations_Manager, I want the Operations Agent to generate a daily morning briefing at 06:00 MST covering today's jobs, resource gaps, and field operations status, so that I can start the day with a clear picture of what needs attention.

#### Acceptance Criteria

1. THE Operations Agent SHALL run automatically every day at 06:00 MST (22:00 UTC previous day) via Celery Beat.
2. WHEN the Operations Agent runs, THE System SHALL query all jobs scheduled for today with status not in `{completed, invoiced, draft}`.
3. FOR each job missing a vehicle assignment or driver assignment, THE System SHALL create an `Agent_Event` with `event_type = 'alert'`, `severity = 'warning'`, and a body identifying the job number, client, and missing resource.
4. FOR each job where `estimated_weight_kg > vehicle.capacity_kg`, THE System SHALL create an `Agent_Event` with `severity = 'warning'` identifying the overload.
5. THE System SHALL count jobs still in `draft` status at 06:00 MST and create a single `Agent_Event` if any exist, warning of potential SLA breaches.
6. THE Operations Agent SHALL generate a 3-sentence morning briefing using the LLM incorporating job counts, resource gaps, and operational field status (compactor overdue count, containers needing pickup, open disruptions, staff on site).
7. THE morning briefing SHALL be persisted as an `Agent_Event` with `event_type = 'report'` and `severity = 'info'`.

### Requirement 7: Fleet & Maintenance Agent — Daily Vehicle Health Check

**User Story:** As an Operations_Manager, I want the Fleet Agent to automatically check vehicle service schedules and fuel consumption daily and alert me when vehicles need attention, so that breakdowns at client sites are prevented.

#### Acceptance Criteria

1. THE Fleet Agent SHALL run automatically every day at 07:00 MST (23:00 UTC previous day) via Celery Beat.
2. WHEN the Fleet Agent runs, THE System SHALL query all vehicles with `next_service_date <= CURRENT_DATE + 14 days` and `status != 'retired'`.
3. FOR each vehicle within 14 days of its service date, THE System SHALL create an `Agent_Event` with `event_type = 'alert'`, `severity = 'warning'`, and a body identifying the vehicle registration, service due date, and current odometer.
4. FOR each vehicle whose service date has passed (`next_service_date < CURRENT_DATE`), THE System SHALL create an `Agent_Event` with `severity = 'critical'`.
5. THE Fleet Agent SHALL also be triggered event-driven when a GPS anomaly is detected via the MQTT gateway (route deviation > 5 km from assigned job zone).
6. THE Fleet Agent SHALL generate a daily fleet health summary using the LLM and persist it as an `Agent_Event` with `event_type = 'report'`.

### Requirement 9: Client Intelligence Agent — On-Demand RAG Query Handling

**User Story:** As a Management user or Operations_Manager, I want to ask the AI assistant natural language questions about any client's history, waste volumes, and service performance, so that I can get instant answers without manually querying the database.

2. THE Client Intelligence Agent SHALL retrieve up to 8 relevant chunks from the Milvus collection before generating a response.

### Requirement 10: Intent Detection and Agent Routing

**User Story:** As a platform user, I want my chat messages to be automatically routed to the most relevant AI agent, so that I receive domain-expert responses without having to select an agent manually.

#### Acceptance Criteria

1. THE System SHALL analyse each incoming chat message using keyword scoring across five intent categories: `compliance`, `esg`, `operations`, `fleet`, and `client`.
2. THE System SHALL route the message to the agent with the highest keyword match score.
3. IF no keywords match any category (all scores = 0), THE System SHALL default to the Client Intelligence Agent.
4. THE intent detection SHALL be case-insensitive and match partial keyword phrases (e.g. "90-day" matches "90 day").
5. THE System SHALL expose the detected intent in the SSE `metadata` event so the frontend can display which agent is responding.

### Requirement 11: RAG Pipeline — Document Ingestion

**User Story:** As a Management user, I want to upload documents to the knowledge base and have them automatically chunked, embedded, and stored in Milvus, so that the AI agents can retrieve relevant content from them.

#### Acceptance Criteria

1. THE System SHALL accept document uploads via `POST /api/v1/ai/ingest-document` for file types: PDF, DOCX, TXT, CSV, Markdown, PNG, JPEG, WEBP, TIFF.
2. THE System SHALL enforce a maximum file size of 50 MB per upload.
3. WHEN a document is uploaded, THE System SHALL save it to disk, create a `Document` record in the database, and queue a Celery task for ingestion.
4. THE ingestion Celery task SHALL extract text from the document, clean it, chunk it into segments of approximately 800 words with 100-word overlap, generate embeddings via Ollama `nomic-embed-text`, and store the chunks in Milvus.
5. WHEN ingestion succeeds, THE System SHALL update the `Document` record with `ingested_into_rag = True` and `ingestion_error = None`.
6. WHEN ingestion fails at any stage, THE System SHALL update the `Document` record with `ingested_into_rag = False` and `ingestion_error` set to a descriptive error message.
7. THE System SHALL support re-ingestion of a failed document via `POST /api/v1/ai/documents/{id}/re-ingest`, which resets the error state and re-queues the ingestion task.
8. ONLY users with roles `management` or `superadmin` SHALL be permitted to upload or delete documents.

### Requirement 12: RAG Pipeline — Context Retrieval

**User Story:** As a platform user, I want the AI assistant to retrieve relevant context from the knowledge base before answering my questions, so that responses are grounded in actual platform documents and regulatory references.

#### Acceptance Criteria

1. WHEN `use_rag = True` in a chat request, THE System SHALL generate a query embedding for the user's message and search the Milvus collection for the top-k most similar chunks (default k=5, configurable up to 20).
2. THE System SHALL inject retrieved chunks into the LLM system prompt under a clearly delimited `--- RELEVANT CONTEXT ---` section.
3. WHEN `client_id` is provided, THE System SHALL filter the Milvus search to return chunks from that client's documents as well as platform-wide documents.
4. THE System SHALL include the number of RAG chunks used in the SSE `metadata` event.
5. IF the Milvus collection is unavailable or the embedding call fails, THE System SHALL degrade gracefully to pure LLM generation without returning an error to the user.
6. THE System SHALL use COSINE similarity as the search metric with `nprobe=16` for the IVF_FLAT index.

### Requirement 13: SSE Streaming Chat

**User Story:** As a platform user, I want AI responses to stream token-by-token in the chat interface, so that I see the response being generated in real time rather than waiting for the full response.

#### Acceptance Criteria

1. THE `POST /api/v1/ai/chat` endpoint SHALL return an `EventSourceResponse` that streams LLM output as Server-Sent Events.
2. THE SSE stream SHALL emit a `metadata` event first, containing the model name, number of RAG chunks used, and client_id scope.
3. THE SSE stream SHALL emit `token` events for each LLM output token, with `{"token": "<text>", "done": false}` as the data payload.
4. THE SSE stream SHALL emit a final `done` event with `{"token": "", "done": true, "total_tokens": <n>, "rag_chunks_used": <n>}`.
5. IF the LLM service is unreachable, THE SSE stream SHALL emit an `error` event with a descriptive message and terminate.
6. IF the LLM request times out (> 120 seconds), THE SSE stream SHALL emit an `error` event and terminate.
7. THE System SHALL support conversation history of up to 20 previous messages injected into the LLM context.

### Requirement 14: Agent Event Persistence and Notification

**User Story:** As an Operations_Manager or Management user, I want all agent-generated alerts and reports to be persisted and surfaced in the platform UI, so that I never miss an important automated finding.

#### Acceptance Criteria

1. THE System SHALL persist every agent-generated alert, recommendation, and report as an `Agent_Event` record in the `agent_events` table with fields: `id`, `agent_name`, `event_type`, `severity`, `title`, `body`, `reference_type`, `reference_id`, `is_read`, `created_at`.
2. THE System SHALL expose `GET /api/v1/ai/agent-events` with filters for `is_read`, `agent_name`, `severity`, `event_type`, and `reference_type`, returning paginated results ordered by unread critical events first.
3. THE System SHALL expose `PATCH /api/v1/ai/agent-events/{id}/read` to mark individual events as read.
4. THE System SHALL expose `POST /api/v1/ai/agent-events/mark-all-read` to bulk-mark all unread events as read.
5. WHEN a `critical` severity event is created, THE System SHALL broadcast it to the WebSocket `agent-alerts` room via `POST /internal/broadcast-alert`.
6. THE `GET /api/v1/ai/agent-events` response SHALL include `unread_count` and `critical_unread_count` summary fields for the notification badge.

### Requirement 15: AgentAlertFeed Frontend Component

**User Story:** As a platform user, I want to see a live feed of agent alerts in the AI Assistant page, so that I am immediately aware of critical findings without navigating away.

#### Acceptance Criteria

1. THE `AgentAlertFeed` component SHALL display a scrollable list of `Agent_Event` records fetched from `GET /api/v1/ai/agent-events`.
2. THE `AgentAlertFeed` SHALL poll for new events every 30 seconds using TanStack Query `refetchInterval`.
3. WHEN a new WebSocket message arrives on the `agent-alerts` channel, THE `AgentAlertFeed` SHALL immediately add the new event to the top of the feed without waiting for the next poll.
4. EACH event in the feed SHALL display: agent name badge, severity badge (colour-coded: red=critical, amber=warning, blue=info), title, truncated body (max 2 lines), and relative timestamp.
5. WHEN a user clicks an event, THE System SHALL mark it as read via `PATCH /api/v1/ai/agent-events/{id}/read` and expand the full body.
6. THE `AgentAlertFeed` SHALL display an unread count badge on the feed header.

### Requirement 16: AgentStatusPanel Frontend Component

**User Story:** As a Management user, I want to see the current status and last-run time of each AI agent, so that I know whether the agents are running correctly.

#### Acceptance Criteria

1. THE `AgentStatusPanel` component SHALL display one status card per agent (Compliance, ESG, Operations, Fleet, Client Intelligence).
2. EACH card SHALL show: agent name, last run timestamp (from the most recent `Agent_Event` for that agent), next scheduled run time, and a status indicator (online/offline based on whether the last run was within the expected interval).
3. THE `AgentStatusPanel` SHALL fetch data from `GET /api/v1/ai/agent-events` filtered by `event_type=report` to determine last-run times.
4. THE `AgentStatusPanel` SHALL include a "Run Now" button for each scheduled agent (Compliance, ESG, Operations, Fleet) that calls the appropriate trigger endpoint.
5. THE "Run Now" button SHALL be restricted to users with roles `management` or `superadmin`.

### Requirement 17: RAG System Health and Status

**User Story:** As a superadmin, I want to monitor the health of the RAG system components (Milvus, Ollama, document ingestion queue), so that I can detect and resolve issues before they affect agent performance.

#### Acceptance Criteria

1. THE `GET /api/v1/ai/rag-status` endpoint SHALL return the status of: Ollama connectivity (reachable/unreachable), Milvus connectivity (reachable/unreachable), Milvus collection row count, number of documents pending ingestion, and number of documents with ingestion errors.
2. THE `GET /health` endpoint SHALL include RAG system status as part of the platform health check.
3. THE AI Assistant page SHALL display a health indicator dot (green=online, red=offline) in the chat header, updated every 60 seconds.

---

### Requirement 18: Role-Based Access Control for AI Features

**User Story:** As a superadmin, I want AI features to be accessible only to users with appropriate roles, so that sensitive operational data surfaced by agents is protected.

#### Acceptance Criteria

1. THE `POST /api/v1/ai/chat` endpoint SHALL be accessible to all authenticated users.
2. THE `POST /api/v1/ai/ingest-document` and `DELETE /api/v1/ai/documents/{id}` endpoints SHALL be restricted to users with roles `management` or `superadmin`; all other roles SHALL receive HTTP 403.
3. THE `GET /api/v1/ai/agent-events` endpoint SHALL be accessible to all authenticated users.
4. THE "Run Now" agent trigger endpoints SHALL be restricted to users with roles `management` or `superadmin`.
5. WHEN a `client` role user submits a chat message, THE System SHALL automatically scope the RAG retrieval to that client's documents only.
